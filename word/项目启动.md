# 齐思 · A自习室 —— 项目启动与运行指南

> 本文档说明如何在本地从零搭建并启动 **齐思（A自习室）** 项目的前端与后端，使其可在浏览器访问。
>
> 适用环境：Windows 11 + Git Bash（命令以 Bash 语法给出，必要时附 CMD 等价写法）。

---

## 🚀 一键启动（脚本，推荐）

完成下方「二、首次部署」后，日常启动可用仓库根目录的 **`start.bat`** 一键拉起全部服务（PostgreSQL / Redis / 后端 / 前端）。

**用法**：双击 `d:\yangtze\project\2026\qisi\qisi_frontend\start.bat`，或在终端执行：

```bat
d:\yangtze\project\2026\qisi\qisi_frontend\start.bat
```

脚本会依次启动下表服务，每个长驻服务开**独立窗口**运行、互不影响，且对**已在运行的服务自动跳过**（可重复执行，不会端口冲突）：

| 步骤 | 服务 | 窗口 / 端口 |
| --- | --- | --- |
| 1 | PostgreSQL | Docker 容器 `qisi-postgres`（:5432，幂等） |
| 2 | Redis | 独立窗口（:6379，带密码） |
| 3 | 后端 Django | 独立窗口（:8001） |
| 4 | 前端 uni-app | 独立窗口（:5273） |

启动完成后浏览器访问：

- **前端**：`http://localhost:5273/`
- **后端 API**：`http://localhost:8001/`

**停止**：关闭对应的命令行窗口即可；PostgreSQL 另执行 `docker stop qisi-postgres`。详见下文「四、停止服务」。

> **使用前提**：已按「二、首次部署」创建好 `venv\`、Docker 容器 `qisi-postgres`、前端依赖；且 Docker Desktop 已运行。脚本用 `%~dp0` 自动定位仓库根目录，可随项目整体移动。

## 二、首次部署（从零搭建）

### 2.1 配置环境变量

仓库根目录已有 `.env`。若没有，从模板复制：

```bash
cp .env.example .env
```

关键配置（开发环境默认值，按需修改）：

```bash
# 数据库（与下方 Docker 容器保持一致）
DB_NAME=qisi_ai_tutor
DB_USER=qisi_admin
DB_PASSWORD=qisi_pg_2026_StrongPwd
DB_HOST=localhost
DB_PORT=5432

# Redis（需带密码，Celery/缓存要用）
CELERY_BROKER_URL=redis://:Redis_2026_StrongPwd@localhost:6379/0
CACHE_URL=redis://:Redis_2026_StrongPwd@localhost:6379/3

# CORS（必须包含前端实际端口 5273）
CORS_ALLOWED_ORIGINS=http://localhost:5173,http://localhost:5273
```

### 2.2 准备 PostgreSQL（Docker 方式，推荐）

启动 Docker Desktop，等待右下角图标变为运行状态后，执行：

```bash
# 拉取镜像
docker pull postgres:16

# 启动容器（库 / 用户 / 密码 / 端口 与 .env 完全一致）
docker run -d --name qisi-postgres \
  -e POSTGRES_DB=qisi_ai_tutor \
  -e POSTGRES_USER=qisi_admin \
  -e POSTGRES_PASSWORD=qisi_pg_2026_StrongPwd \
  -p 5432:5432 \
  postgres:16

# 验证就绪
docker exec qisi-postgres pg_isready -U qisi_admin -d qisi_ai_tutor
# 期望输出：... accepting connections
```

> 容器创建后只需一次。后续重启电脑后用 `docker start qisi-postgres` 即可，无需再 `run`。

### 2.3 创建后端虚拟环境并安装依赖

```bash
cd d:/yangtze/project/2026/qisi/qisi_frontend

# 1) 创建虚拟环境（位于仓库根目录 venv/）
python -m venv venv

# 2) 安装依赖（直接用 venv 内的 python，无需激活）
venv/Scripts/python.exe -m pip install --upgrade pip
venv/Scripts/python.exe -m pip install -r requirements.txt
```

> **已知仓库缺陷（如重新克隆需注意）**：
>
> - `requirements.txt` 中 `aliyun-sdk-oss` 是错误包名，PyPI 上无可用版本；代码实际 `import oss2`，应改为 **`oss2>=2.18,<3.0`**。
> - 代码使用了 `pydantic`（`apps/parser/schemas/`），但 `requirements.txt` 漏写，需补 **`pydantic>=2.0,<3.0`**。
>
> 以上两处**已在当前仓库修复**；若从原始仓库重新拉取，请按上述补回。

### 2.4 执行数据库迁移

```bash
venv/Scripts/python.exe manage.py migrate
```

看到各 app 的 `... OK` 即成功。（可选）创建后台超级用户：

```bash
venv/Scripts/python.exe manage.py createsuperuser
```

### 2.5 安装前端依赖

```bash
cd d:/yangtze/project/2026/qisi/qisi_frontend/uniapp
npm install
```

## 三、日常启动（每次开发）

按顺序启动 4 个服务。建议每个服务开一个终端窗口。

### 3.1 启动 PostgreSQL

```bash
docker start qisi-postgres
```

### 3.2 启动 Redis（需带密码，与 .env 一致）

```bash
redis-server --port 6379 --requirepass "Redis_2026_StrongPwd"
```

> 验证：另开终端 `redis-cli -a "Redis_2026_StrongPwd" ping` → 应返回 `PONG`。
> 若 Redis 作为 Windows 服务已配置好，也可用 `net start Redis`（需管理员权限）。

### 3.3 启动后端（Django）

```bash
cd d:/yangtze/project/2026/qisi/qisi_frontend
venv/Scripts/python.exe manage.py runserver 0.0.0.0:8001
```

> **必须先启动 PostgreSQL**，否则 runserver 在启动阶段的 migration 检查会因连不上 5432 而崩溃退出。
>
> 如需让 AI 试卷解析等异步任务可用，**另开终端启动 Celery Worker**（见第五节）。

### 3.4 启动前端（uni-app H5）

```bash
cd d:/yangtze/project/2026/qisi/qisi_frontend/uniapp
npm run dev:h5
```

启动后控制台会输出：

```text
➜  Local:   http://localhost:5273/
```

在浏览器打开 **`http://localhost:5273/`** 即可。

## 五、可选：启动 Celery（AI 解析等异步任务）

试卷的 AI 解析走异步流水线（11 步），不启动 Celery 会导致解析任务一直卡住。开发时如需测试解析功能：

```bash
cd d:/yangtze/project/2026/qisi/qisi_frontend
venv/Scripts/python.exe -m celery -A config worker -l info --pool=solo
```

> Windows 下 Celery prefork 池有限制，建议加 `--pool=solo`（或 `--pool=gevent`）。
> `.env` 中 `CELERY_TASK_ALWAYS_EAGER=False`（已设为异步），生产/正常开发不要改回 True。

## 六、常见问题排查

**1. 后端启动报 `connection to server at "localhost", port 5432 failed: Connection refused`**
→ PostgreSQL 没启动。先 `docker start qisi-postgres`，确认 `pg_isready` 通过再启动后端。

**2. `pip install` 报 `No matching distribution found for aliyun-sdk-oss`**
→ requirements.txt 用了错误包名。改 `aliyun-sdk-oss>=3.18,<4.0` 为 `oss2>=2.18,<3.0`（当前仓库已修复）。

**3. 后端 `ModuleNotFoundError: No module named 'pydantic'`**
→ 补装 `venv/Scripts/python.exe -m pip install "pydantic>=2.0,<3.0"`（当前仓库 requirements 已含）。

**4. 前端启动后访问 5173 打开的是另一个项目（maintain3.0）**
→ 端口串台。本项目前端固定 **5273**，请访问 `http://localhost:5273/`。若端口被改回 5173，检查 `uniapp/vite.config.ts` 的 `server.port` 与 `uniapp/src/manifest.json` 的 `h5.devServer.port` 是否均为 5273。

**5. 前端页面能打开，但调用接口报网络错误 / 跨域**
→ 确认后端在 8001 已启动；前端 `/api`、`/media` 经 Vite 代理转发到 `http://localhost:8001`（见 `uniapp/vite.config.ts`）。`.env` 的 `CORS_ALLOWED_ORIGINS` 必须包含 `http://localhost:5273`。

**6. Redis 报 `NOAUTH` 或 `WRONGPASS`**
→ Redis 启动时的 `--requirepass` 必须与 `.env` 中 `Redis_2026_StrongPwd` 完全一致。
