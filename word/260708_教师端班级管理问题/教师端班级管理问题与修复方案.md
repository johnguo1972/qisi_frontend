# 教师端班级管理问题与修复方案

> 日期：2026-07-08
> 范围：**仅教师端班级管理** —— `apps/institutions/class_views.py` 中的 `GET /api/v1/classes`（列表）与 `POST /api/v1/classes`（创建）。
> 硬约束：修复**只动 `class_views.py` 这一个文件的两个函数 + 顶部 import**；不改模型、不改迁移、不改学生端、不动其它端代码与数据。
> 说明：本文档**只描述问题与方案，不修改代码**，由实施者按方案落地。

---

## 0. 背景（为什么发现这两个问题）

现象：教师账号 `15883633570` 在教师端 `GET /api/v1/classes` **看不到任何班级（返回空）**，但学生端 `GET /api/v1/student/my-classes` 能看到班级。

DB 实证（只读排查）：
```
TEACHER id=1 role=teacher (15883633570)
TOTAL  class=1   class_teacher=0   class_student=1
CLASS id=1  creator=15883633570  in_class_teacher=False  class_teachers=[]  active_students=1
```
- 班级 `id=1` 的 `creator_teacher` 是该教师，但 **`class_teacher` 关联表为空（0 条）**。
- 教师端列表只认 `class_teacher` 表 → 空；学生端列表认 `class_student` 表 → 有。

> 结论：当前这一条班的直接成因是**数据问题**（`class_teacher` 缺失），非代码功能 bug。
> 但代码存在 **2 个隐患**，让这类数据问题一旦发生就**无兜底、难排查**。本文档针对这 2 个隐患给出修复方案。

---

## 问题 1：教师端班级列表不认 `creator_teacher`（设计不一致）

### 位置
- 文件：`apps/institutions/class_views.py`
- 函数：`_class_list_impl`（第 99–126 行）
- 关键代码（第 101–103 行）：
  ```python
  qs = Class.objects.filter(
      class_teachers__teacher=request.user,
  ).order_by('-created_at')
  ```

### 现象
列表查询**只通过 `ClassTeacher` 关联表**过滤，**完全不看 `Class.creator_teacher` 字段**。
因此，任何「有 `creator_teacher`、但 `ClassTeacher` 缺失」的班级，其创建者在教师端**永远不可见**。

### 影响
- **可见性丢失**：历史数据、手动入库、早期版本（当时建班逻辑尚未写 `ClassTeacher`）创建的班，创建者教师在自己班级列表里看不到，无法管理（改班、看学生、发任务等）。
- **无兜底**：即便教师是 `creator_teacher`，也没有任何 fallback 让他看到自己创建的班。
- **本次现象的放大器**：正是这个隐患让 `class_teacher` 一旦缺失，创建者就彻底失去对班的可见性。
- **波及面**：仅教师端列表展示；不影响学生端（学生端走 `ClassStudent`，独立查询）。

### 解决方案（仅改 `_class_list_impl` 的 filter 一处 + 顶部补 import）

(a) 文件顶部 import 区（第 3 行 `import uuid` 附近）补充：
```python
from django.db.models import Q
```

(b) 第 101–103 行改为（增加「或 creator_teacher 是当前用户」分支，并 `distinct()` 去重）：
```python
qs = Class.objects.filter(
    Q(class_teachers__teacher=request.user) | Q(creator_teacher=request.user),
).distinct().order_by('-created_at')
```

### 为什么这样改是安全的
- 查询条件**只增不减**：原来能看到的班（有 `ClassTeacher`）仍然命中第一条分支，行为不变、无回归。
- 新增的 `creator_teacher` 分支只让「缺失 `ClassTeacher` 但自己是创建者」的班多一层兜底可见性。
- `distinct()` 消除 `class_teachers` 多对多 join 产生的重复行。
- `creator_teacher` 允许为 null（`SET_NULL`）；null 不匹配 `=request.user`，无副作用。

## 问题 2：创建班级两步操作未包在事务里（健壮性）

### 位置
- 文件：`apps/institutions/class_views.py`
- 函数：`_create_class_impl`（第 38–78 行）
- 关键代码（第 66–71 行）：
  ```python
  cls = serializer.save()
  # Create ClassTeacher relation
  ClassTeacher.objects.create(
      class_obj=cls, teacher=request.user, role='owner',
  )
  ```

### 现象
「建 `Class`」与「建 `ClassTeacher`」是两条独立 SQL，中间**没有 `transaction.atomic()`**。

### 影响
- **脏数据风险**：若第二步 `ClassTeacher.objects.create(...)` 因任何原因失败（数据库抖动、唯一约束冲突、连接中断等），第一步已落库的 `Class` **不会回滚**，从而留下**正好就是「有 `Class` 无 `ClassTeacher`」的脏数据**——即本次现象那种状态。
- 概率虽低，但一旦发生，就会触发问题 1 的不可见陷阱，且无任何兜底。
- **波及面**：仅教师端「创建班级」接口；不影响学生端、不影响阶段1 已完成的 F1 信号（F1 基于 `ClassStudent` 的 `post_save`，与建班事务无关）。

### 解决方案（仅用事务包裹原有两步 + 顶部补 import）

(a) 文件顶部 import 区补充：
```python
from django.db import transaction
```

(b) 第 66–71 行改为（用 `with transaction.atomic():` 包住两步，逻辑完全不变）：
```python
with transaction.atomic():
    cls = serializer.save()
    # Create ClassTeacher relation
    ClassTeacher.objects.create(
        class_obj=cls, teacher=request.user, role='owner',
    )
```

### 为什么这样改是安全的
- 两步的执行顺序、参数、返回值**全部不变**；只是保证原子性。
- 第二步失败时，第一步的 `Class` 自动回滚，不再产生脏数据。
- 对正常成功路径零影响（成功时事务正常提交，行为与原先一致）。

## 修复后验证清单（教师端，用真实/临时数据自测）

> 前置：一个教师 token，至少 1 个机构 + 该教师是其 `teacher` 成员。

1. **正常建班不回归**
   `POST /api/v1/classes` 建一个班 → `GET /api/v1/classes` 能看到该班（问题 2 事务不影响成功路径）。
2. **creator 兜底可见（问题 1）**
   在 DB 删掉某班的 `ClassTeacher` 记录（保留 `creator_teacher`）→ `GET /api/v1/classes` **仍能看到**该班（修复前看不到）。
3. **事务回滚（问题 2，可选压测）**
   构造 `ClassTeacher.objects.create` 失败的场景（如临时把 `class_teacher` 表锁/约束制造冲突）→ 确认 `Class` **未落库**（修复前会落库成脏数据）。
4. **学生端不回归**
   `GET /api/v1/student/my-classes` 返回与修复前**一致**（学生端代码未动）。
5. **分页/状态过滤不回归**
   `GET /api/v1/classes?status=active&page=1&page_size=20` 行为正常（`distinct()` 后分页计数仍准确）。

针对当前 `class id=1`（`creator_teacher=15883633570`，`ClassTeacher` 缺失）：

  INSERT INTO class_teacher (class_id, teacher_id, role) VALUES (1, 1, 'owner');

二选一即可。推荐 B（根治 + 对所有历史班生效）。
